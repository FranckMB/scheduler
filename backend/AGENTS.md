## Zone gotchas (facts not in the root docs)

1. **All commands run in the php-fpm container** — `backend/Makefile` wraps `docker compose exec`. `composer`/`php bin/console` on the host fail.
2. **CS-Fixer rule set** is `@Symfony` + `@Symfony:risky` + `@PHP84Migration` + `@PHP80Migration:risky` (see `.php-cs-fixer.dist.php` — the canonical source).
3. **Entities use string UUIDs** (no auto-increment) and scalar FK columns (`clubId`, `seasonId`… as strings, no object associations).
4. **`declare(strict_types=1)`** in every file; single-action controllers use `#[AsController]` + `__invoke()`.
5. **Redis lock** `ClubGenerationLock` (SETEX NX + release token) serialises generation per club.
6. **Cache pool `cache.schedule`** — dedicated Redis pool for engine payloads, TTL 4h, invalidated by `CacheInvalidationListener`.
7. **`SOFT_LOCK_PENALTY = 10_000`** in `ScheduleConstraintBuilder` — builder-side weight sent to the engine for soft-locked slots (distinct from the engine's own tier weights).
8. **Test DB required** before PHPUnit: `make db-init-test` once, then `make phpunit` (`--group phase1`).
9. **Stuck-schedule watchdog** `app:schedules:reconcile-stuck` (BCK-01) must run on a **cron in prod** (e.g. every 10 min): it fails `GENERATING` schedules abandoned by a crashed/OOM worker. Terminal-status nets are detailed in root `docs/project-map.md` §2.4.
9b. **Period-reminder cron** `app:periods:remind` (cockpit palier C) must run on a **daily cron in prod** (e.g. @8h): it emails a club's managers (owner+admin) when a period starts in exactly 14/7/3 days and still has NO overlay plan (`--dry-run` / `--date=YYYY-MM-DD` for rehearsal). Never auto-acts (mirrors the in-app radar). Needs `FRONTEND_BASE_URL` (empty → email omits the cockpit link). Same club-walk + per-club GUC pattern as the BCK-01 watchdog; managers resolved via `ClubUserRepository::findManagementEmails` (raw DBAL, works with an empty GUC). Mailer is **sync**.
10. **Dated constraints excluded from generation** — a `Constraint` with `calendarEntryId` set belongs to a `CalendarEntry` period (cockpit, palier A) and MUST NOT feed base-plan generation. Generation/validation/report load constraints via `ConstraintRepository::findPermanentByClubSeason` (`calendarEntryId IS NULL`), never `findByClubSeason`. See `specs/evolution/accueil-cockpit-temporel.md` §9ter.
11. **Sticky cockpit unlock** — `Season.socleValidatedAt` is stamped once when the season's baseline schedule is first `VALIDATED` (or a `VALIDATED` schedule is set as baseline); **never reset** (reopen keeps it). Exposed on `/api/me`. Gates the frontend cockpit vs work-loop.
12. **School holidays = global reference** — `school_holiday_period` has **no club_id / no RLS** (public national data, like `sport`). Seeded from a versioned JSON (`app:school-holidays:seed`, idempotent by `(zone, holidayType, schoolYear)`) — **no network at runtime**. Read via `GET /api/school-holidays` (filtered by the club's `schoolZone` + a window). `Club.schoolZone` is derived from the FFBB code by `SchoolZoneResolver` at register (`app:clubs:backfill-school-zone` for existing clubs, dry-run by default). FFBB code = **3-letter league + 4-digit zero-padded department + sequence** (e.g. `GES0067060` → 67 Strasbourg zone B; `GUY0973021` → 973 Guyane, a DOM → no A/B/C zone → null). DOM/TOM and non-matching shapes → null → manually editable `schoolZone`.
13. **Cockpit conflicts** — `GET /api/calendar-entries/{id}/conflicts`: for a period entry, reads its closed venues from dated FACILITY constraints, then the **baseline** schedule's slot templates on those venues whose ISO weekday (`ScheduleSlotTemplate.dayOfWeek` = **1=Mon..7=Sun**) falls in the window → returns concrete conflicting dates. Palier A **surfaces** conflicts, moves nothing (overlay = palier B).
14. **Period overlays (palier B)** — a `Schedule` with `calendarEntryId` set is the OVERLAY of a period (inverse of `CalendarEntry.overlayScheduleId`, stamped server-side at POST). Generation branches on it: `GenerateScheduleHandler` calls `ScheduleConstraintBuilder::buildForOverlay` (closure = permanent + dated constraints; holiday = dated only + full structure), which **BYPASSES the schedule-input cache** and **skips `assignBaselineIfFirst`** (an overlay never becomes the season baseline). Deletion of an overlay/period goes through `OverlayManager` (slots + diagnostics have no FK cascade — purge them). ⚠ **`config.type=venue_closed` (the dated constraint the cockpit creates) is a NO-OP in the engine** — `buildForOverlay` expands each closed venue into per-team FACILITY HARD `config.forbiddenVenueId` constraints, which the engine honors (`constraints.py` → `forbidden_assignments`). If the dated constraint ever needs to be consumed as-is → engine change → re-validate with the user.
15. **Destructive baseline reopen** — reopening the season BASELINE while overlays exist → **409 `{code:"overlays_exist", count, overlays[]}`**; the client re-POSTs `{"confirmDeleteOverlays": true}` → `OverlayManager` deletes the overlays (entries + dated constraints KEPT — the period falls back to "signalée"). Guarded by `ReopenScheduleTest`.
